# RAVEN Tool Inventory

> **This file is the single source of truth for the server/tool catalog, the read/write split, and the environment variables.**
> `README.md`, `docs/SYSTEM_DESIGN_AND_ARCHITECTURE.md`, and `.github/copilot-instructions.md` link here instead of repeating these facts, so the numbers can only live in one place.
>
> **Maintenance:** the catalog block below (between the `GEN` markers) is **generated** by `scripts/gen-inventory.mjs` from each server's registered tools and their `readOnlyHint` annotations. After changing tools, run `npm run gen-inventory`; CI (`npm run gen-inventory:check`) fails any PR where it is stale or a tool lacks an explicit read/write hint. The surrounding prose (this header, the env-var table) is hand-maintained.
>
> **Read/write convention:** "write" means the tool mutates the connected upstream or creates protected local output as part of an operational workflow. Jenkins config exports, Jenkins artifact downloads, and `artifactory_download_artifact` are conservatively marked write because they create local files; `clone_repo` retains its historical read classification because it does not change the Bitbucket remote.

### Remote proxy (data egress)

#### jarvis-mcp — dynamic remote proxy — **data egress**
`jarvis-mcp` registers **zero tools in source**. It is a pass-through proxy: it forwards `tools/list`, `tools/call`, prompts, and resources straight to the remote Jarvis MCP endpoint (`JARVIS_BASE_URL`, default `https://jarvis-api.example.gov.bc.ca/mcp`) with an `Authorization: Bearer <JARVIS_TOKEN>` header (`packages/jarvis-mcp/src/index.ts`, `config.ts`).

- The tool list is **whatever the remote advertises at runtime** — currently ~6: `get_application`, `list_ministries`, `search_applications`, `list_technologies`, `get_application_provenance` (read) and `update_application` (write). Treat that count as a snapshot, not a fixed contract.
- **This is the only server that proxies MCP traffic to a remote endpoint.** Every RAVEN server makes outbound calls to its upstream; what's unique to Jarvis is that it forwards opaque MCP requests/responses to a separately-hosted BC Gov API whose tool surface is defined remotely, rather than running a fixed local tool set. Relevant to the "local-only" framing and to FOIPPA/data-residency review — see `SYSTEM_DESIGN_AND_ARCHITECTURE.md`.

## How this was derived

The catalog block above (between the `GEN` markers) is generated by [`scripts/gen-inventory.mjs`](../scripts/gen-inventory.mjs). It spins each server up over an in-memory MCP transport, calls `tools/list`, and reads each tool's `readOnlyHint` annotation for the read/write split — so the catalog cannot disagree with the registered tools.

```bash
npm run build                 # the generator imports each server's built dist/server.js
npm run gen-inventory         # rewrite the GEN block in this file
npm run gen-inventory:check   # CI gate: fails if the block is stale or any tool lacks a read/write hint
```

`jarvis-mcp` is the exception — it registers no tools in source (it proxies a remote surface), so its row is described from a constant in the generator. The env-var table above is still hand-maintained; cross-check it with `grep -rhoE 'process\.env(\.[A-Z_0-9]+|\["[A-Z_0-9]+"\])' packages/*/src scripts`.
